# 基盤文法 — パラダイム中立な形式骨格

> contingency-dsl 基盤層（Ψ）の一部。パラダイム中立な CFG / LL(2) 構造、メタ文法、トークン型、構文木の規約、識別子の規約を定義する。オペラント固有のスケジュール産出規則は `operant/grammar.md` に、レスポンデント固有の産出規則は `respondent/grammar.md` に置く。

**関連文書:** [LL(2) 形式的証明](ll2-proof.md) — FIRST₁/FIRST₂/FOLLOW₂ 集合、LL(2) 構文解析表、曖昧性のなさの証明、パラダイム保存の証明。この文法には LL(2) 決定点が 1 箇所しか存在しない（オペラント合成の `arg_list` 中の `PosTail` のみ）。他のすべての決定点は LL(1) である。

---

## 1. 設計原則

DSL 文法は、基盤層レベルで次の 4 つの基準を満たす。

1. **表記の忠実性**: 行動分析文献の慣習と一致すること（パラダイム固有の形式は `operant/grammar.md` および `respondent/grammar.md` に置く）。
2. **構文解析の曖昧性のなさ**: すべての式は一意な構文木を持つ。
3. **合成可能性**: 任意に入れ子になった式をサポートする。
4. **Python への埋め込み性**: 単独のテキスト形式としても、Python のコンストラクタ呼び出しとしても利用可能。

これらの原則はパラダイム中立である。オペラントのスケジュール式、レスポンデントのプリミティブ、合成手続き、実験層のいずれにおいても成立する。

## 2. メタ文法（パラダイム中立な骨格）

以下の BNF 断片は、対象パラダイムがオペラント（三項随伴性）かレスポンデント（二項随伴性）かに**依存しない**部分である。パラダイム固有の産出規則（`ScheduleExpr`、`PairExpr` 等）は、各層の仕様でこの骨格を拡張する。

```bnf
<file>          ::= <experiment> | <program>

<program>       ::= <program_annotation>* <param_decl>* <binding>* <expr>
<expr>          ::= <paradigm_specific_expr>        -- 各層で定義

<binding>       ::= "let" <ident> "=" <expr>

<param_decl>    ::= <param_name> "=" <value>
                  -- パラダイム固有キーワードは、対応する層の文法で列挙する
                  -- （例: LH, COD, FRCO は operant/grammar.md に属し、
                  --  @iti, @cs_interval は annotations/extensions/respondent-annotator
                  --  に属する）。

<value>         ::= <number> <time_unit>?
<time_unit>     ::= "s" | "ms" | "min"
<number>        ::= [0-9]+ ("." [0-9]+)?

<ident>         ::= [a-z_][a-zA-Z0-9_]*   -- <reserved> に一致してはならない
```

**識別子の命名制約.** 識別子は ASCII の小文字またはアンダースコア（`[a-z_]`）で始まり、それに続く文字は任意の ASCII 文字・数字・アンダースコアでなければならない。これにより、DSL のキーワードおよびスケジュール型プレフィックス（すべて大文字で始まる）と語彙的に重ならないことが保証される。各層は `<reserved>` を自層のキーワード集合で拡張する。全層にわたる和集合は `<ident>` の正規集合と非交差である。

**将来予約.** `def` は、パラメータ付きテンプレート導入時の識別子衝突を防ぐため、基盤層レベルで予約されている。`architecture.md §4.4` を参照。

## 3. トークン型

字句解析器は以下のトークン分類を認識する。パラダイム層は追加トークン（例: スケジュール・プレフィックス `FR`、プリミティブ・コンストラクタ `Pair.ForwardDelay`）を導入するが、下記は基盤レベルのものである。

| 分類 | 例 | 備考 |
|---|---|---|
| 数値リテラル | `5`, `30.0`, `0.003` | §2 の `<number>` |
| 時間単位サフィックス | `s`, `ms`, `min` | 連結または空白区切り |
| 識別子 | `baseline`, `treatment_1` | `[a-z_]` で始まる |
| 大文字識別子 | `Baseline`, `Phase1` | フェーズ名。`[A-Z]` で始まる |
| キーワード | `let`, `def`, `phase`, `use` | 予約語 |
| 区切り記号 | `(` `)` `,` `=` `->` | 構造記号 |
| 文字列リテラル | `"Long-Evans"` | ダブルクォート。注釈値用 |
| 注釈マーカー | `@` | 注釈の付与を導入する |

## 4. 構文木の規約

DSL はソーステキストを以下の 3 フェーズで抽象構文木（AST）へコンパイルし、各フェーズは JSON Schema により規定される。

| フェーズ | 名称 | 型の宇宙 | スキーマ | 主要な性質 |
|---|---|---|---|---|
| フェーズ 1 | 展開前 | `IdentifierRef`, `RepeatModifier` を含む | `ast-parsed.schema.json` | 未解決の参照を含みうる |
| フェーズ 2 | 展開後 | `IdentifierRef`, `RepeatModifier` を除く | `ast-resolved.schema.json` | 全参照が解決され、マクロ糖衣も展開済み |
| フェーズ 3 | 解決済み | フェーズ 2 + パラダイム層のデフォルト伝播 | `ast-resolved.schema.json` | プログラムレベルのデフォルトが伝播済み |

各フェーズ遷移は全域である。すなわち、対象フェーズの型付けされた AST を生成して成功するか、意味エラーを報告して失敗するかのいずれかであり、部分的な AST は返されない。

この 3 フェーズ・パイプラインはパラダイム中立であり、オペラントとレスポンデントの両層が一様に使用する。パラダイム固有のデフォルト伝播（例: `operant/theory.md §1.6.1` のオペラント層 LH 伝播）はフェーズ 2 → フェーズ 3 で作用する。

## 5. 表記の柔軟性（基盤レベル）

基盤レベルの文法では、同一の値に対して複数の等価な記述形式を許容する。どの原子がこれらの形式を使用するかはパラダイム層が規定し、基盤層は語彙形式そのものだけを固定する。

```
<number>-<unit>        -- JEAB 現代標準（ハイフン区切り。例: "60-s"）
<number> <unit>        -- 空白区切り単位（例: "60 s"）
<number><unit>         -- 連結単位（例: "60s"）
<number>               -- 裸の数値、単位は文脈で暗黙的に補う
```

上記の 4 形式は AST レベルで同一の `(value, unit)` 対に解決される。構文解析器は区切り方に依らず `<value>` を一様に読む。

## 6. 構文糖衣（基盤レベル）

基盤レベルには汎用の糖衣が 2 種類存在する。パラダイム層は、これに加えて自層固有の糖衣を追加できる。

**`let` 束縛** はマクロ展開（テキスト的な置換）である。束縛名は `<ident>` を満たさねばならない。すなわち小文字またはアンダースコアで始まり、予約語のいずれにも一致してはならない。

```
let baseline = <expr_1>
let treatment = <expr_2>
<expr_using baseline, treatment>
```

は、対応する let を含まない構文木へ展開される。`let` は可変状態、クロージャ、再帰的定義を一切導入しない。展開後は、基本 CFG 内の let を含まない構文木となる。

**スコープ規則.** `let` 束縛は次の 3 つの意味制約に従う。

1. **シャドウイング禁止.** 束縛名はプログラム内で一意でなければならない。名前の重複は静的エラーである（例: `"duplicate binding 'x' (first defined at line N)"`）。
2. **前方参照禁止.** 束縛 `let x_k = e` において、`e` に出現するすべての識別子は `{x_1, ..., x_{k-1}}`、すなわち先行する束縛に定義されているものでなければならない。未定義もしくは後方定義の識別子を参照することは静的エラーとなる。系として、推移的な相互参照（例: `let a = b; let b = a`）は拒否される。
3. **トップレベルのみ.** 束縛は `<program>` レベルにのみ現れる。`let` はパラダイム固有の合成式の内部には現れない。

実装は加えて、未使用の束縛（メイン式または後続の束縛から一度も参照されない束縛名）に対して警告を発することが推奨される。

**`def`（予約）.** `def name(args) = <expr>` の形のパラメータ付きテンプレートは、将来の設計チェックポイントに向けて予約されている。キーワードは識別子衝突を防ぐため現時点で予約されている。導入後、`def` は呼び出しグラフ上の DAG 正当性条件に従ってマクロ置換として展開される（`architecture.md §4.4` 参照）。

## 7. エラー回復方針（パラダイム中立）

適合構文解析器は、どのパラダイムを解析しているかに関わらず、次のエラー回復方針に従わねばならない（MUST）。

**エラー報告要件（MUST）:**
- エラーコード、位置（行および列）、エラー分類（`LexError`, `ParseError`, `SemanticError`）を含む、少なくとも 1 件のエラーを報告する。
- エラーコードは、パラダイム固有のエラーレジストリ（`conformance/foundations/errors.json`、`conformance/operant/errors.json`、`conformance/respondent/errors.json`）で定義されたコードのいずれかに一致しなければならない。

**複数エラーの回復（SHOULD）:**
- 構文解析器は **panic-mode recovery**（Aho et al., 2006, §4.8.2）を実装することが推奨される。エラー検出時、同期トークンに到達するまで入力トークンを読み捨て、その後解析を再開する。
- 同期トークン: `)`, `,`, 改行（`\n`）, `EOF`。
- 検出したすべてのエラーを位置順のリストとして報告する。

**部分 AST（MUST NOT）:**
- 構文解析器は、エラーが存在する場合に部分的な AST を返してはならない。エラー時の結果は AST を伴わないエラーリストである。

**エラー数の上限（MAY）:**
- 構文解析器は、連鎖的なエラーノイズを防ぐため、設定可能な最大エラー件数（推奨デフォルト: 10）に達した時点で停止してもよい。

**理由.** 「1 エラーで停止」を強制すると、単一の誤配置文字が下流の問題を隠してしまう DSL ではユーザに受け入れられない。panic-mode 回復は、マルチエラー報告を可能にする最も単純な戦略であり、句レベル回復やエラー産出規則による回復の複雑さを回避できる（Aho et al., 2006, §4.8）。

## 8. LL(2) 性（基盤レベル）

上記の文法骨格は、すべての決定点で LL(1) である。DSL 全体で唯一の LL(2) 決定点はオペラント合成の `arg_list`（`PosTail` の曖昧性解消）内部に存在し、これはオペラント層にローカルであって基盤骨格を汚染しない。形式的証明は `ll2-proof.md` を参照。
