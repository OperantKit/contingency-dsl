# 相変化基準

> contingency-dsl 実験層の一部。相変化基準のカタログ — あるフェーズが終了し次のフェーズが開始する宣言的条件 — を規定する。基準集合は列挙された非 TC ファミリである: `Stability`, `FixedSessions`, `PerformanceCriterion`, `CumulativeReinforcements`, `ExperimenterJudgment`。このカタログを越えるランタイム条件付き基準（例: 反応率ベースのスケジュール切替）は実験層ではなく `contingency-core` に属する。

**関連文書:**
- [実験 / フェーズ列](phase-sequence.md) — 基準が付与される場所。
- [実験 / 文脈](context.md) — 文脈変化との基準相互作用。
- スキーマ: `schema/experiment/phase-sequence.schema.json` — 基準形状定義の権威ソース。

## 1. 列挙された基準ファミリの根拠

Sidman (1960) は、実験行動分析における相転移決定の適切な基礎は定常状態論理であると論じる: 相転移は、任意に外部から課された時間的制約ではなく、行動が安定した再現可能なレベルに達することに依存すべきである。JEAB および関連 EAB 出版物は一般にこの慣習に従うが、消去相、プローブ／テスト相、応用研究設定では時折非安定性基準を要することを認める。

下記の基準ファミリは標準的な JEAB 慣習を列挙する。このファミリは閉じている（追加的変更は design-philosophy §8.1 に従う）。かつ宣言的である — 各基準は解析時のリテラル・パラメータで完全に仕様化され、プログラムはセッションにわたって関連測度を追跡して評価する。いかなる基準も任意のランタイム計算を許容しない。

## 2. `Stability` — 定常状態基準

JEAB 基礎研究で最も一般的な基準である。指定した行動測度が指定した連続セッション数にわたって指定した許容値未満で変動したとき、フェーズは終了する。

```
Stability(window_sessions=5, max_change_pct=10, measure="rate", min_sessions=20)
```

**パラメータ:**
- `window_sessions` — 変動性を評価する連続セッション数。2 以上でなければならない。
- `max_change_pct` — 窓にわたる測定変数の最大変化率。`10` は「≤ 10% の変動」を意味する。
- `measure` — 評価する行動測度。列挙: `rate`（単位時間あたりの反応数）、`iri`（強化間時隔）、`latency`（反応潜時）、`reinforcers`（セッションあたりの強化子数）。
- `min_sessions` — 安定性チェックが成功しうる前の、フェーズ内の最小総セッション数。初期変動性が人為的理由で低い場合に早期の相変化を防ぐ。

**JEAB 表現例.** 「反応率が連続 5 セッションで 10% 未満変動したとき、最低 20 セッションを条件として、安定性と判定した。」

**引用.** Sidman, M. (1960). *Tactics of scientific research*. Basic Books.

## 3. `FixedSessions` — 固定セッション数

フェーズは行動に関わらずちょうど `count` セッション継続する。消去相（減衰の安定性が目標でない）やプローブ／テスト相（固定数のテスト試行が規定される）で一般的。

```
FixedSessions(count=10)
```

**パラメータ:**
- `count` — フェーズ内のセッション数。

**JEAB 表現例.** 「この相は 10 セッション継続した。」

## 4. `PerformanceCriterion` — 行動依存閾値

指定した測度が指定した方向で閾値を越えたとき、フェーズは終了する。弁別訓練と応用研究で一般的。

```
PerformanceCriterion(measure="rate", threshold=1.0, op="<", window_sessions=3)
```

**パラメータ:**
- `measure` — 行動測度。列挙: `rate`, `count`, `latency`, `iri`。
- `threshold` — 数値閾値。
- `op` — 比較演算子: `<`, `<=`, `>`, `>=`。
- `window_sessions` — 基準が満たされるべき連続セッション数。デフォルトは 1（単一セッションで十分）。

**JEAB 表現例.** 「反応が連続 3 セッションで毎分 1 反応未満に落ちるまで。」

## 5. `CumulativeReinforcements` — 総強化子閾値

セッションにわたって累積強化子数が呈示された後、フェーズは終了する。訓練時間を時間ではなく強化経験に対して調整する必要があるとき一般的。

```
CumulativeReinforcements(count=300)
```

**パラメータ:**
- `count` — フェーズ内の全セッションにわたる総強化子数。

**JEAB 表現例.** 「300 回の強化子が得られるまで訓練を継続した。」

## 6. `ExperimenterJudgment` — 裁量的

フェーズは実験者の裁量で終了する。形式的基準はない。古い JEAB 文献と、プロトコル設計が裁量的決定を許容する応用場面で一般的。

```
ExperimenterJudgment
```

**パラメータなし.**

**JEAB 表現例.** 「相変化は実験者の裁量で行われた。」

**注記.** `ExperimenterJudgment` は真正な JEAB 慣習であるため許容される。推奨されるからではない。この基準で手続きを符号化することは、相転移の裁量性を DSL で明示化することであり、それ自体が方法論的透明性の一形式である。

## 7. 終端フェーズの規約

`PhaseSequence` の終端（最終）フェーズは **基準を持たない**。実験は終端フェーズの完了で終わる。終端フェーズに基準フィールドを持つプログラムに遭遇した場合、検証エラーを発するべきである。これはスキーマレベルで強制される（`schema/experiment/phase-sequence.schema.json`）。

## 8. カタログが閉じている理由

実験層は設計上、非 TC である（design-philosophy §2, architecture.md §4.1）。任意のランタイム計算基準 — 例えば、切替規則を伴う反応率の連続監視に依存する基準 — を許容することは、実験層をチューリング完全領域に押し込む。そのような動的スケジュール遷移は、明示的に TC であり姉妹パッケージとして存在する `contingency-core` に属する。実験層は標準 JEAB 慣習を宣言的に扱う。これらの慣習を越えるものは `contingency-core` にエスカレートする。

このカタログに追加基準を加えることは、design-philosophy §8.1 の追加的手続きを通じて可能である: 追加は既存の基準意味論を変更してはならず、セッションごとの測度上の宣言的述語として仕様化可能でなければならない。
